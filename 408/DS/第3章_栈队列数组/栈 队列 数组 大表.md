### 🚨 架构师的红笔预警区 (Exception List)

**[🔥 Fatal_1：顺序栈的 ++ / -- 顺序导致数据覆盖/野指针]**
*   **坑点定位**：408 填空题、代码大题逻辑错乱的万恶之源。
*   **大白话批注**：
    *   当 `top = -1` 时：先移指针再干活！（`++top`入栈，先挪位置再放数据；出栈反之，先拿数据再`top--`撤退）。
    *   当 `top = 0` 时：先干活再移指针！（`top++`入栈，先在空位放数据，再挪向下一个空位；出栈反之，先退一格`--top`，再拿数据）。
    *   **一字口诀：初始为-1就前缀(++top)，初始为0就后缀(top++)。千万别混用！**

**[🔥 Fatal_2：GetTop (读栈顶) 时的幽灵越界修改]**
*   **坑点定位**：手写 `GetTop` 伪代码时，顺手写成 `return S.data[S.top--]` 或 `return S.data[--S.top]`。
*   **大白话批注**：`GetTop` 是只读接口 (Read-Only)！绝对、绝对不允许出现 `++` 或 `--` 改变栈指针状态。遇到 `top=0` 的约定，读取元素必须是 `S.data[top-1]`，**注意是 `top-1` 而不是 `--top`！**

**[💥 Error_1：共享栈的“撞车”判定边界 (极易搞错下标)]**
*   **坑点定位**：考研选择题/大题直接问共享栈判满条件。
*   **大白话批注**：共享栈满 **绝对不是** `top0 == top1`！因为它们存放的都是有效元素，真相等的话意味着两个数据存在了同一个内存格子里，早就发生内存踩踏 (Memory Corruption) 了。
*   **红笔写死**：**相邻即满**：`top0 + 1 == top1`。

**[⚠️ Warning_1：出栈和退队后，原数据的物理残留]**
*   **坑点定位**：选择题考查内存状态。
*   **大白话批注**：不管是数组还是链表，执行了出栈操作，其实只是 `top` 指针回退了，或者节点被游离了。**物理内存里的那个值(x)还在那里！**（直到下一次 Push 被覆写或被 free 回收）。不要认为出栈后那个格子里的数据就变成 `NULL` 或清空了！

### 🚨 架构师的红笔预警区 (Queue Exception List)

**[🔥 Fatal_1：链队出队时的“尾指针悬空 (Dangling Pointer)”]**

- **坑点定位**：手写链队 Dequeue 出队代码时，极易漏写边界条件。
- **大白话批注**：
  - 出队操作都是在 front 那头搞事，本来跟 rear 没半毛钱关系。但是！当队列里**只剩最后一个有效节点**时（此时 front->next == rear），你把它 free 掉了！
  - 这就导致 rear 指向了一块被回收的垃圾内存！如果不拉回来，下次执行入队 rear->next = p 时，**程序当场段错误 (Segmentation Fault) 崩溃！**
  - **红笔写死**：**“带头结点链队出队，必须判断 if(front->next == NULL) rear = front;！不带头结点，必须判断 if(front == rear) { front=NULL; rear=NULL; }”**

**[🔥 Fatal_2：循环队列 tag 方案的 Length 计算陷阱]**

- **坑点定位**：选择题考你 tag 方案的队列长度。
- **大白话批注**：
  - 很多人死背公式 (rear - front + MaxSize) % MaxSize。
  - 但这公式在 tag 方案里有个致命缺陷：**当队列满的时候，front == rear，你套这个公式算出来的长度是 0！** 满队列长度怎么可能为 0？！
  - **红笔写死**：**“Tag 方案算长度，先看头尾等不等！若等，看 Tag 定生死（Tag=0 长度0，Tag=1 长度 MaxSize）。若不等，再套老公式！”**

### 🚨 架构师的红笔预警区 (Matrix Exception List)

**[🔥 Fatal_1：题干中的“下标魔法” (Zero-Indexed Trap)]**
*   **坑点定位**：408 命题组最爱玩的阴招。上面那张表，全天下所有的辅导书都是基于**“矩阵行号列号从 1 开始”**推导的。但 408 考卷上，经常冷不丁来一句：**“设矩阵 $A[0 \dots n-1][0 \dots n-1]$...”**
*   **大白话批注**：
    *   一旦矩阵的 $i, j$ **从 0 开始**了，你脑子里的所有公式立刻全部作废！
    *   **千万别去脑补新公式！用【面积法】现场算！**
    *   例如：对称矩阵，求 $(i,j)$。因为从 0 开始，所以第 0 行有 1 个，第 1 行有 2 个... 它上方有 $i$ 行（从 0 到 $i-1$），所以上方总面积是 $\frac{i(i+1)}{2}$。在本行，它前面有 $j$ 个人。
    *   **红笔写死防坑定理**：**无论怎么变，下标 $k$ 永远等于【排在它前面的总元素个数】！只要画个图，把上面有几个框，左边有几个框加起来，就是 $k$！**

**[🔥 Fatal_2：列优先的逆向绞杀 (Column-Major)]**
*   **坑点定位**：题目突然说“按列优先存储下三角矩阵”。
*   **大白话批注**：
    *   “列优先”就是把矩阵转个 90 度来看。
    *   如果你硬背公式，遇到列优先直接崩溃。
    *   **面积法秒解**：找元素 $(i,j)$ 在**列优先**下前面的元素。
    *   上方变成**左侧**！它左边有 $j-1$ 个完整的列！这 $j-1$ 列是一个**倒梯形**（第1列有 $n$ 个，第2列有 $n-1$ 个...），用等差数列算面积。然后再算在本列它上方有几个元素 ($i-j$ 个)。加起来就是答案！

### 📦 附加沙盒：【稀疏矩阵 (Sparse Matrix) 防御沙盒】
*(如你之前所问，稀疏矩阵自成一派，我们用一个微型沙盒把它封死)*

**核心考点只有两句话，请用紫笔抄下：**
1. **压缩原理**：只存非零元素。彻底失去随机存取（$O(1)$）能力。
2. **底层存储选型**：
    *   **三元组 (顺序表)**：存 `(行i, 列j, 值v)`。**缺点**：矩阵做加法/乘法时，如果要插入新非零元素，必须移动大量数据。
    *   **十字链表 (链表)**：每个非零结点包含 5 个域：`row, col, value, right, down`（向右连同行，向下连同列）。**优点**：完美解决矩阵运算时插入/删除非零元素的问题！

**[🔥 Fatal_1：丧失 O(1) 随机存取特性 (Random Access Loss)]**
*   **坑点定位**：408 概念判断题。
*   **大白话批注**：不管是普通多维数组，还是刚才讲的特殊矩阵（对称、三角），它们压缩后依然保留了 **“代数映射”**，所以你想找 $A[i][j]$，套个公式 $O(1)$ 瞬间就拿到了。

**[🔥 Fatal_2：三元组表的有序性前提]**
*   **坑点定位**：三元组的内部排列规则。
*   **大白话批注**：三元组在物理数组里存放时，**绝对不是乱放的**！它必须严格按照 **“行优先”（或列优先）** 的顺序排列。即：先按 `row` 从小到大排，`row` 相同的再按 `col` 从小到大排。打乱了就无法正确复原矩阵！
