# 串

### ⬛ 状态 2：【生成表结构 (Schema Generation) - 全量数据版】

#### 📄 表1：【模式匹配引擎底层逻辑对比表】

| 对比维度                | BF 算法 (Brute-Force) 暴力引擎                               | KMP 算法 状态机引擎                                          |
| :---------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **最坏时间复杂度**      | $\mathbf{O(n \times m)}$                                     | $\mathbf{O(n + m)}$                                          |
| **空间复杂度**          | $\mathbf{O(1)}$ (无需辅助空间)                               | $\mathbf{O(m)}$ (需开辟 next/nextval 数组)                   |
| **主串指针 `i` 轨迹**   | **疯狂回溯**：每次失配，`i` 退回到 `i - j + 2`               | **永不回溯**：单调递增，一路向前扫描                         |
| **模式串指针 `j` 轨迹** | **清零重置**：每次失配，`j` 退回到 `1`                       | **智能跳转**：每次失配，`j` 退回到 `next[j]`                 |
| **408 命题雷区**        | **退化场景**：主串 `000...001`，模式串 `0001` 时，时间复杂度拉满，极其低效。 | **优势场景**：仅在主/模串存在大量**“部分匹配”**时，KMP 才有绝对优势；若首字母就不匹配，BF 也很快。 |

#### 📄 表2：【KMP 状态机核心数组对照表】

| 对比维度                               | `next` 数组 (基础版路由表)                                   | `nextval` 数组 (终极优化路由表)                              |
| :------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **物理定义**                           | 记录模式串失配点**前**的子串中，“最长相等前后缀”的长度 + 1。指示 `j` 的下一跳。 | 状态机的路径压缩版：消除了 `next` 数组中“明知必定失败还要跳”的冗余状态。 |
| **推导依赖**                           | 仅依赖**模式串自身**的字符排布。                             | 必须依赖**模式串** + 已经求出的 **`next` 数组**。            |
| **优化痛点**                           | 如果跳转后的字符和当前失配字符**一模一样** ($P_j == P_{next[j]}$)，跳过去必然再次失配，浪费一次 CPU 周期。 | 完美填补痛点，若发现跳转后字符一样，直接继承其再下一跳的路由（即让 $nextval[j] = nextval[next[j]]$）。 |
| **408 默认初始值** <br>*(下标从1开始)* | 无脑写死：<br>$\mathbf{next[1] = 0}$<br>$\mathbf{next[2] = 1}$ | 半写死：<br>$\mathbf{nextval[1] = 0}$<br>$nextval[2]$ 取决于 $P_1$ 是否等于 $P_2$。 |
