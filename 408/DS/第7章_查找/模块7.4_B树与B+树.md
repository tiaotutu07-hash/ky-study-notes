# 模块 7.4:B 树与 B+ 树 —— 磁盘时代的多路平衡

> **本模块全景导图**
>
> 回到 7.3 结尾留下的钩子——**红黑树用"适度平衡"换来了更低的维护成本,是 BST 的工程级优化**。但红黑树解决的是**内存中**的查找问题。当数据规模大到内存装不下、必须存到磁盘上时,另一种矛盾浮现:**每一次结点访问都是一次磁盘 I/O,代价巨大**。二叉树再矮也是 $\log_2 n$ 层,100 万数据要 20 层——就是 20 次 I/O,太慢。能不能让树**"更矮更胖"**?让每个结点**装很多关键字**,一次 I/O 读一大块,把树高压到 3-4 层?这就是 B 树的动机,也是磁盘时代的**多路平衡查找树**。
>
> 本模块两个主角:
> ① **B 树**(也叫 B-树,"多路平衡查找树")—— 每个结点最多 m 棵子树、m-1 个关键字;所有叶结点(失败结点)在同一层;高度永远保持在 $O(\log_m n)$ 这一更矮的量级。408 考最细:**5 条性质、高度推导上下界、插入分裂、删除 3 情况 + 合并连锁**。
> ② **B+ 树** —— B 树的**数据库专用变体**。所有数据只放在叶层,非叶结点只做"索引的索引";叶层还串成链表,既能**多路查找**又能**顺序扫描**。MySQL、Oracle、PostgreSQL 的主键索引几乎全用 B+ 树。
>
> 学完本模块你将能独立应对七类 408 高频题型:**B 树定义判别**(2009)、**5 条性质计算关键字/结点数**(2013/14/18/21)、**B 树高度上下界推导**、**插入分裂构造**(2020)、**删除三情况**(2012/22)、**合并连锁**(2023)、**B+ 树应用与 B vs B+ 差异**(2016/17)。
>
> 核心脉络是 **"平衡力度滑尺"** 在 7.3 基础上的**再进化**:BST → AVL → 红黑树(内存,二叉多层)→ **B 树(磁盘,多路少层)→ B+ 树(数据库索引)**。维度从 "是否严格平衡" 切换到 "每一层装多少",整个查找章节在这里达到高潮。

## 一、从内存到磁盘 —— B 树的登场

这一节本身不展开算法,只做"动机塑造"——把本模块主角放进 408 整个查找家族的演进链里,同时回答一个关键问题:**为什么我们还需要 B 树?**

### 1. 为什么二叉树不够用了?—— I/O 代价的挑战

还记得吗,7.3 结尾我们把树形查找三剑客 BST / AVL / 红黑树做了总结,三者都能做到 $O(\log_2 n)$ 级查找。这听起来很好——100 万数据只要 20 次比较。但这个结论有一个**隐藏前提**:数据全在**内存**里。

工程现实里,很多数据量级远超内存——数据库表动辄上亿条、文件系统索引项可能有几十万个、搜索引擎的倒排索引更是吓人。这些数据只能存在**磁盘**上,查找时按需加载到内存。**磁盘 I/O 的代价,远远超过内存访问**:

| 操作 | 典型延迟 | 相对代价 |
|---|---|---|
| CPU 寄存器访问 | ~1 ns | 1× |
| 主存访问 | ~100 ns | 100× |
| 机械磁盘随机 I/O | ~10 ms | **10,000,000×** |
| SSD 随机 I/O | ~100 μs | 100,000× |

**关键洞察**:用二叉树查找 100 万数据,虽然只要 20 次**比较**,但每次比较都可能跨越一次磁盘 I/O——总耗时 20 × 10ms = **200 毫秒**,用户会明显感觉卡顿。所以,在磁盘场景下,比较次数不重要,**磁盘 I/O 次数才是性能瓶颈**。

### 2. 多路平衡的核心思想:矮胖换扁平

既然 I/O 次数等于**树的高度**,我们自然想到一个办法——**把树压矮**。怎么压?**让每一个结点装更多关键字**。

```
【二叉树的"瘦高"】               【多路树的"矮胖"】
        [x]                          [k1 k2 ... km-1]
       /   \                        /   |   \    ...   \
     [x]   [x]             [...]  [...] [...]  [...]  [...]
     / \   / \              /|\   /|\   /|\   /|\
   ....................
每层 2 叉 → 20 层          每层 m 叉 → 3-4 层(m=100)
20 次 I/O                   3-4 次 I/O
```

把 2 路换成 m 路(m 可以是几十、几百甚至上千),树高从 $\log_2 n$ 下降到 $\log_m n$。举例,100 万数据:
- **二叉树**:$\log_2 10^6 \approx 20$ 层 → 20 次 I/O
- **3 阶 B 树**:$\log_3 10^6 \approx 13$ 层 → 13 次 I/O
- **100 阶 B 树**:$\log_{100} 10^6 = 3$ 层 → **3 次 I/O**

效率提升 **6 倍以上**。

**结点内部怎么用?** 读入一个结点(一次 I/O)后,它在内存里就是一个有序数组,内存里可以用**顺序查找**或**折半查找**快速定位,然后决定走哪一条子树分支继续下一次 I/O。

### 3. B 树的定位:磁盘上的动态查找

B 树(**B-Tree**,"B-"读作"B 减",但**不能读成 B 减树**,连字符仅是连词)由 Bayer 和 McCreight 于 1972 年提出,是为**磁盘/文件系统查找**而设计的数据结构。王道原话:

> 所谓 **m 阶 B 树**是所有结点**平衡因子均等于 0** 的 **m 路平衡查找树**。

这句话有三层含义:
- **m 路**:每个结点最多 m 个孩子
- **平衡查找树**:查找效率稳定,左右子树高度差有约束
- **平衡因子均等于 0**:所有叶结点在**同一层**(比 AVL 的"差 ≤ 1"更严格)

**跨章关联**:这里的"平衡因子"概念直接继承自 7.3 的 AVL,但含义升级——AVL 是"任意结点左右子树高度差 ≤ 1",B 树是"**任意结点左右(所有)子树高度差 = 0**"。B 树对平衡的要求**比 AVL 更严**。这是因为磁盘场景下,高度任何一点增加都要付出 I/O 代价,必须保持绝对整齐。

思想已经定好(矮胖换扁平),接下来 B 树是如何用 5 条性质把这个思想固化下来的?这就进入本模块最重的一节。

## 二、B 树 —— 多路平衡查找树

### 1. B 树的阶数 m(2009 真题考点)

**阶数** m 是 B 树最核心的参数,指 **B 树中孩子结点数的最大值**,通常用字母 m 表示。

一棵 m 阶 B 树的每个结点:
- **最多** 有 m 个孩子(m 棵子树)
- **最多** 有 m − 1 个关键字

这是因为 n 个关键字会把结点内部切成 n + 1 段区间,对应 n + 1 棵子树。反过来,**n 个子树对应 n − 1 个关键字**——这是 B 树的基本代数关系,背下来。

⚠️ **易错点**:阶数 m 永远是 B 树能容纳的**孩子数上限**,**不是**关键字数。3 阶 B 树每个结点最多 3 个孩子、最多 2 个关键字;5 阶 B 树每个结点最多 5 个孩子、最多 4 个关键字。2009 真题的陷阱就在这里。

### 2. B 树的五条核心性质(2013/14/18/21 真题高频)

一棵 m 阶 B 树**或为空树,或满足下列特性的 m 叉树**:

| 性质 | 内容 | 命题追踪 |
|---|---|---|
| **①** | 树中每个结点**至多**有 m 棵子树,**至多**有 m − 1 个关键字 | 2013/2014/2018 |
| **②** | 若根结点不是叶结点,则**至少**有 2 棵子树(即至少 1 个关键字) | 2021 |
| **③** | 除根结点外的所有非叶结点至少有 ⌈m/2⌉ 棵子树,即至少有 **⌈m/2⌉ − 1** 个关键字 | 2013 |
| **④** | 所有非叶结点的结构如下:`n \| P₀ \| K₁ \| P₁ \| K₂ \| P₂ \| ... \| Kₙ \| Pₙ`,其中 K 递增,Pᵢ 所指子树关键字夹在 Kᵢ 和 Kᵢ₊₁ 之间 | — |
| **⑤** | 所有**叶结点**(失败结点 / 外部结点)在**同一层**,并且**不带信息** | 王道独家定义 |

⚠️ **术语地雷(必看)**:B 树的"**叶结点**"是指**虚构的失败结点、外部结点**,**不带信息**!这与普通二叉树(7.3)里"没有孩子的真实结点"的术语**完全不同**,也与后面 B+ 树里"存数据的那一层"的术语**完全不同**。王道原话:

> 所谓**叶结点**,指这些结点实际上并不存在,指向这些结点的指针为空。

王道书脚注还特别提醒:"多数教材将 B 树的叶结点定义为失败结点,本书也采用这种定义,但 **408 真题将 B 树的叶结点定义为最底层的终端结点**。" 做题时遇到"B 树叶结点",要**看清语境**——多数情况下指失败结点,408 真题里指最底层的真实结点(也叫**终端结点**)。

### 3. 结点结构详解

B 树结点的典型结构,是 B 树所有操作的"原子单位":

```
┌─────┬────┬────┬────┬────┬────┬────┬─────┬────┬────┐
│  n  │ P₀ │ K₁ │ P₁ │ K₂ │ P₂ │ ... │ Kₙ │ Pₙ │
└─────┴────┴────┴────┴────┴────┴────┴─────┴────┴────┘
   ↑           ↑                         ↑
   │         关键字 key                  │
  关键字                                  │
  个数                               指针 point
                                  指向子结点

• n   = 该结点当前的关键字个数
• Kᵢ  = 第 i 个关键字,且 K₁ < K₂ < ... < Kₙ(严格递增)
• Pᵢ  = 指针,指向子树的根
• Pᵢ₋₁ 所指子树内所有关键字 < Kᵢ < Pᵢ 所指子树内所有关键字
```

**关键代数关系**:**#Pᵢ(子树数)= #Kᵢ(关键字数)+ 1**。牢记这个"**子比键多一**"关系——它是 B 树所有计算题的基础,也是 B+ 树差异对比的关键参考点。

```c
// B 树结点结构体定义(王道风格)
#define MAXM 20                              // B 树的最大阶数
typedef struct BTNode {
    int keynum;                              // 结点中关键字的个数
    struct BTNode *parent;                   // 指向父结点的指针
    KeyType key[MAXM];                       // 关键字数组,key[1..keynum] 有效
    struct BTNode *ptr[MAXM];                // 子树指针数组,ptr[0..keynum] 有效
} BTNode, *BTree;
```

### 5. B 树的查找(两基本操作)

**B 树的查找**与二叉排序树类似,但有一个关键区别:每个结点不是两路分支,**而是根据该结点的子树所做的多路分支决定**。

B 树的查找操作包含**两个基本动作**:
1. **在 B 树上找结点**(走到哪个结点)—— 因为 B 树常存储在磁盘上,**这个操作是在磁盘上进行**
2. **在结点内找关键字**(在该结点的关键字数组里定位 key)—— 找到目标结点后,**先将结点读入内存,再在内存内用顺序查找或折半查找**

因此,磁盘上进行查找的次数 = 目标结点在 B 树上的层数,这决定了 B 树的查找效率。

**查找算法伪代码**:

```c
// 在 B 树 T 上查找关键字 key,成功返回指向结点的指针 + 结点内位置
BSTNode *B_Search(BTree T, KeyType key){
    BTree p = T;                             // p 指向当前正在比对的结点
    while(p != NULL){
        int i = 1;
        while(i <= p->keynum && key > p->key[i])  // 在结点内顺序查找
            i++;
        if(i <= p->keynum && p->key[i] == key)    // 找到
            return p;                        // 返回结点指针
        else
            p = p->ptr[i-1];                 // 未找到,走入对应子树(一次磁盘 I/O)
    }
    return NULL;                             // 查找失败
}
```

#### 【例题 7.4.2 查找 46 —— 失败】

```

        内存内比较:33 < 46 → 走 P₂,进入右子树
步骤 2(磁盘 I/O #2):读 [48]
        内存内比较:46 < 48 → 走 P₀,进入左子树
步骤 3(磁盘 I/O #3):读 [45, 47]
        内存内比较:45 < 46 < 47 → 走 P₁(应到下一层叶结点)
步骤 4:叶结点是虚构失败结点(指针为空)→ 查找失败
```

**注意**:这里"到了叶结点"意味着走到了"本该指向第 4 层的指针,但该指针为 NULL"——即 B 树查找失败的判定。

⚠️ **易错点**:B 树的查找**可能在非叶结点就找到**(如查 15 在第 3 层就命中,但 15 并非失败叶)。这与后面 B+ 树"查找必须到叶才能成功"**完全不同**。做题时看清题意,别把两种树的行为混淆。

### 6. B 树的高度上下界

B 树的大部分操作所需的**磁盘存取次数**与 B 树的高度成正比。因此分析 B 树的高度范围是性能分析的核心。

**前置说明**:B 树的高度**不包括失败结点那一层**(王道采用这一定义;少数教材包含)。若一棵 B 树包含 n 个关键字、高度 h、阶数 m,则:

#### 最低高度(每结点塞满)

每个结点最多有 m 棵子树、m − 1 个关键字。高度为 h 的 m 阶 B 树中,关键字数上限满足:

$$
n \le (m-1)(1 + m + m^2 + \cdots + m^{h-1}) = (m-1) \cdot \frac{m^h - 1}{m-1} = m^h - 1
$$

由此反解:

$$
\boxed{h \ge \lceil \log_m(n+1) \rceil = h_{\min}}
$$

#### 最高高度(每结点塞最少)

若每个结点关键字个数达到最少(除根外,每个非叶结点至少 ⌈m/2⌉ 棵子树、⌈m/2⌉−1 个关键字):

```
level 1:   至少  1 个结点
level 2:   至少  2 个结点(根非叶则至少 2 子)
level 3:   至少  2⌈m/2⌉ 个结点
level 4:   至少  2⌈m/2⌉² 个结点
......
level h+1: 至少  2⌈m/2⌉^{h-1} 个**失败(外部)结点**
```

对关键字数为 n 的 B 树,失败结点数为 **n + 1**(这是 B 树的重要结论,与二叉排序树判定树一致)。因此:

$$
n + 1 \ge 2 \lceil m/2 \rceil^{h-1}
$$

反解得:

$$
\boxed{h \le \lfloor \log_{\lceil m/2 \rceil}\!\tfrac{n+1}{2} \rfloor + 1 = h_{\max}}
$$

**跨章关联**:这里的推导思想与 7.3 AVL 最少结点数 $n_h$ 递推**完全同源**——都是通过"**最坏平衡情况下的最少结点数**"反推树的最大高度上界。7.3 是给定高度推结点数,本节是给定关键字数推高度,互为倒推。

#### 【例题 7.4.3 王道例题:3 阶 B 树 8 个关键字的高度范围】

> 假设一棵 3 阶 B 树共有 8 个关键字,其高度范围是多少?

**推演**:$m=3$,$n=8$。
- $h_{\min} = \lceil \log_3 9 \rceil = 2$
- $h_{\max} = \lfloor \log_{\lceil 3/2 \rceil} (9/2) \rfloor + 1 = \lfloor \log_2 4.5 \rfloor + 1 = 2 + 1 = 3$

(精确值 $\log_2 4.5 ≈ 2.17$,向下取整 2,加 1 得 3。王道给出的数值 3.17 是没向下取整直接算出的,结论一致。)

**所以 $2 \le h \le 3$**,即这棵 B 树最矮 2 层、最高 3 层。

### 7. B 树的插入(2020 真题考点)

与二叉排序树的插入相比,B 树的插入**复杂得多**。在 B 树中查找到插入的位置后,并不能简单地将其添加到终端结点中(最底层的非叶结点)——因为此时可能会导致整棵树不再满足 B 树定义中的要求。

B 树的插入分**三步**:

1. **定位**:利用查找算法,从根开始找到**插入位置**(一定是最底层的**终端结点**)。由于查找 key 时会落到失败结点,**插入位置就是失败结点的上一层**——即终端结点中的某个位置
2. **插入**:在该终端结点的关键字有序数组中,按大小插入 key
3. **判断是否分裂**:
   - 若插入后该结点关键字数 ≤ m − 1,**合规,结束**
   - 若插入后该结点关键字数 = m(溢出),**必须分裂**

**分裂规则**(重要):取一个新结点,将溢出结点从中间位置 **⌈m/2⌉** 切开:
- **左部** 的 ⌈m/2⌉ − 1 个关键字保留在原结点
- **右部** 的剩余关键字存入新结点
- **中间位置** ⌈m/2⌉ 的那个关键字**上提**到父结点

若此时父结点关键字个数也超了上限,则**继续分裂**,直至到根。若根也需要分裂,则根上提生成新根,**树高 +1**。

⚠️ **易错点**:分裂的中间位置是 **⌈m/2⌉**(向上取整),**不是 m/2**。对于 3 阶是第 2 个,4 阶是第 2 个,5 阶是第 3 个。考试时先把 ⌈m/2⌉ 算清楚,再看分裂哪一个。

#### 【例题 7.4.4 简单插入 —— 插入 13】

以我们的 3 阶 B 树为例,任务:**插入关键字 13**。

```
Step 1(定位):查找 13 → 从根 18 → 比较 18>13 走左 → 12 → 12<13 走右 → [15] → 15>13 → 走左为空
              ★ 插入位置 = 结点 [15] 的前面
Step 2(插入):把 13 放进 [15] → 得到 [13, 15]
Step 3(检查):[13, 15] 有 2 个关键字,符合 1≤keys≤2 的条件 → 不分裂,结束
```

**结果**(局部变动):

```
              ┌────┐
              │ 12 │
              └┬──┬┘
             ┌┘   └┐
           ┌─┴┐   ┌┴────┐
           │10│   │13 15│   ← 插入成功,无分裂
           └──┘   └─────┘
```

#### 【例题 7.4.5 连锁分裂 —— 插入 22(零壹独家 10 帧演示,2020 真题类型)】

基于同一棵 3 阶 B 树,任务:**插入关键字 22**(此时原树中 [20, 21] 是左中子叶,已经 2 keys 满了)。

**Step 1 定位**:查找 22 → 从根 18 → 18<22 走中 → [23, 30] → 22<23 走左 → [20, 21] → 22>21 走右,插入位置在 [20, 21] 右侧

**Step 2 插入**:把 22 放进去 → 得到 [20, 21, 22]

**Step 3 检查**:[20, 21, 22] 有 3 个关键字,超了 1≤keys≤2 的上限,**必须分裂!**

**Step 4 分裂策略**(核心!):中间位置 ⌈3/2⌉=2 的关键字是 **21**,上提到父结点:

```
分裂前:                 分裂后(本结点):
[20, 21, 22]           左 [20]   右 [22]
                       中间的 21 上提到父 [23, 30]
```

父结点变成 [21, 23, 30] → 又超了 1≤keys≤2!→ **继续分裂父结点**

**Step 5 父结点分裂**:中间位置 ⌈3/2⌉=2 的关键字是 **23**,上提到根:

```
父结点 [21, 23, 30] 分裂:
左 [21]    右 [30]
中间的 23 上提到根 [18, 33]
```

根结点变成 [18, 23, 33] → 又超了 1≤keys≤2!→ **继续分裂根结点**

**Step 6 根分裂**:中间位置的 **23** 上提,形成**新根** [23]。原根分成 [18] 和 [33] 两个新结点。**树高 +1!**

**最终结果**:

```
                        ┌────┐
                        │ 23 │              ← 新根(原树高 +1)
                        └─┬──┘
                  ┌──────────────┐
                ┌─┴──┐        ┌──┴──┐
                │ 18 │        │ 33  │
                └┬──┬┘        └┬──┬─┘
         ┌──────┘   └──┐  ┌───┘   └───┐
       ┌─┴─┐        ┌──┴┐┌┴─────┐   ┌─┴──┐
       │ 12│        │ 21││ 30   │   │ 48 │
       └┬┬─┘        └┬──┘└┬──┬──┘   └┬──┬┘
      ┌─┘└─┐      ┌──┘ ┌──┘  │   ┌───┘  └─┐
    ┌─┴┐ ┌─┴┐  ┌──┴─┐┌─┴─┐  ┌┴──┐┌┴─────┐┌┴────┐
    │10│ │15│  │20 22│  24│  │31 ││45│47││50 52│
    └──┘ └──┘  └─────┘└───┘  └───┘└─────┘└─────┘
```

**关键洞察**:这个 10 帧演示揭示了 B 树最重要的动态性质——**B 树通过分裂的连锁反应来保持绝对平衡,最终可能导致树高增加**。王道静态图只给了单结点分裂的 1 张图,零壹的连锁动画让你看清**分裂如何传递到根**。这也是 B 树与 AVL 树的本质区别——AVL 通过**旋转**调整(局部),B 树通过**分裂 + 上提**调整(可传递到根)。

**跨章关联**:插入连锁分裂与 7.3 AVL 的"调整仅一次"形成鲜明对比。AVL 插入最多一次旋转,而 B 树插入最多可以一路分裂到根。这是**"多路"换来的代价**——每层的维护动作更多,但换来了更低的树高。

### 8. B 树的删除(2012/22/23 真题考点)

B 树的删除**比插入更复杂**,不仅要保证删除后关键字个数 ≥ ⌈m/2⌉ − 1,还可能涉及结点的**合并**。删除分两大情景:

#### 情景 A:被删关键字在非终端结点

若被删关键字 k 不在终端结点(即在内部结点),则**不能直接删除**——删除后那个位置会留下断裂的子树指针。此时可以**用 k 的直接前驱 k′ 或直接后继 k″ 替代 k**,然后在相应的终端结点中删除 k′ 或 k″。由于**前驱/后继必定落在某个终端结点中**,问题就转化为"在终端结点删除关键字"的情况。

> **前驱** = k 左侧子树中"最右下"的关键字
> **后继** = k 右侧子树中"最左下"的关键字

##### 【例题 7.4.6 非终端删除 —— 删 48(零壹演示)】

对主角 3 阶 B 树,**48 是第 2 层的关键字(非终端)**。48 的前驱是什么?48 所在结点的左子树是 [45, 47],其"最右下"元素是 **47**。

**转化**:用 47 替代 48,现在变成 "在终端结点 [45, 47] 中删除 47"。

```
删除前:                      替代后:
        [48]                         [47]
       /    \                       /    \
   [45 47]  [50 52]             [45]    [50 52]
              (此时问题:在终端 [45] 中删 47,转为情景 B)
```

接下来就按"情景 B:终端结点删除"处理。

#### 情景 B:被删关键字在终端结点

此时有 **3 种情况**,判别条件是"被删关键字所在结点的关键字个数"与 ⌈m/2⌉ 的关系:

| 情况 | 判别条件 | 处理方式 | 动作 |
|---|---|---|---|
| **① 直接删** | 所在结点 keys ≥ ⌈m/2⌉ | 直接删除该关键字,不破坏 B 树 | 原地修改 |
| **② 兄弟够借** | 所在结点 keys = ⌈m/2⌉ − 1,且相邻左/右兄弟 keys ≥ ⌈m/2⌉ | **父子换位法** | 父下、兄上 |
| **③ 兄弟不够借** | 所在结点 keys = ⌈m/2⌉ − 1,且相邻左/右兄弟 keys 均 = ⌈m/2⌉ − 1 | **合并**(兄弟 + 双亲的分隔 key) | 合并至一个结点 |

##### 【例题 7.4.7 情况 ① 直接删 —— 删 21(零壹演示)】

在主角 3 阶 B 树中删除 **21**(21 所在结点是 [20, 21],有 2 keys,2 ≥ ⌈3/2⌉=2,满足直接删条件)。

```
删除前:                删除后:
   ... [23, 30] ...         ... [23, 30] ...
        /  |  \                    /  |  \
    [20,21] [24] [31]          [20] [24] [31]   ← 21 直接删,无调整
```

**1 步完成,零代价**。这是最轻松的删除情况。

##### 【例题 7.4.8 情况 ② 兄弟够借 · 父子换位法 —— 删 24(零壹独家详解)】

在主角 3 阶 B 树中删除 **24**(24 所在结点是 [24],1 key,但其左兄弟 [20, 21] 有 2 keys ≥ ⌈3/2⌉=2,**兄弟够借**)。

**核心动作:父子换位法**
- **① 父结点的分隔 key 下移**(补到 24 的空位)
- **② 兄弟结点的最大值上移**(补到父结点被下移的位置)

具体动画:

```
Step 1 删 24:             Step 2 父下 + 兄上(父子换位):
   [23, 30]                      [21, 30]       ← 父 23→21(兄最大上移)
    /  |  \                       /  |  \
[20,21][  ][31]               [20] [23] [31]    ← 兄去最大,空位塞父下移的 23
```

最终结果:父结点 `[21, 30]`、左子 `[20]`、中子 `[23]`、右子 `[31]`。所有结点关键字数都符合 1 ≤ keys ≤ 2。

**为什么叫"父子换位"?** 因为**父 23 变成了子 23**(下移),**兄弟 21 变成了父 21**(上移),父与子的位置做了一次"错层交换"。这样做保证了:
- 父结点的新 key 21 仍然 > 左子 [20] 所有 key、< 右子 [23] 所有 key(B 树性质保持)
- 兄弟结点从 2 keys 变 1 key,仍 ≥ ⌈m/2⌉−1=1(合规)
- 空缺结点获得 1 个 key(仍 ≥ ⌈m/2⌉−1=1,合规)

这一招是王道原文里**被一笔带过**的"父子换位法",零壹 PPT 用完整动画展示了它。**408 大题如果考 B 树删除,这是最容易画错的地方**,务必多练几遍。

##### 【例题 7.4.9 情况 ③ 兄弟不够借 · 合并 —— 删 31(零壹演示)】

在接着上一例的结果上删除 **31**(31 所在结点 [31],1 key;左兄弟 [23] 也只有 1 key,**不够借**)。

**核心动作:合并**
- 将**父结点中分隔 31 所在子树和兄弟子树的那个 key** 下移
- 下移的 key 与**兄弟结点的所有 key 和被删结点的剩余 key** 合并到一个结点里

具体动画:

```
Step 1 删 31 后:          Step 2 合并(父的 30 下借 + 兄弟 [24] + 剩余空):
   [21, 30]                     [21]             ← 父少一个 key(30 下移)
    /  |  \                     /  \
[20] [23] [ ]             [20]  [23, 24, 30]  ← 兄弟 + 父 key + 空 合并
                                                 (这里是极端示例用了三合一,
                                                  正常是 兄弟 + 父 key 合并)
```

**修正说明**:实际上在标准合并中,合并的是**被删结点(空)+ 父分隔 key + 相邻兄弟**。对这个例子更准确的动画:

```
Step 1 删 31:           Step 2 合并 ([24] + 下借 30 + 空):
   [21, 30]                  [21]
    /  |  \                  /  \
[20] [24][ ]             [20]  [24, 30]    ← 合并结果:[24, 30]
```

父结点从 2 keys 减到 1 key,变成 [21]。**父结点仍满足 1 ≤ keys ≤ 2,不触发连锁**。

⚠️ **易错点**:情况 ③ 合并是**兄弟和被删结点合并成一个新结点**,**父结点的分隔 key 夹在中间下移合并**。很多同学容易以为"兄弟替代被删结点",这是错的。记住:**合并 = 兄弟 + 父 key + (被删结点剩余)**,结果是一个更大的结点。

#### 【例题 7.4.10 合并连锁至根 —— 删 10(2023 真题型,零壹独家)】

最后一个高价值例子:**删除 10**,此时 10 所在结点 [10] 只有 1 key,兄弟 [15] 也只有 1 key,**兄弟不够借**,必须合并。但合并后父结点的 key 数会减 1,有可能触发**向根连锁的合并**。

基于上一例结果(删 31 后的形态):

```
开始形态(继续之前的基础上):
             ┌────────┐
             │ 18 │ 33│
             └┬───┬───┬┘
           ┌──┘   │   └──┐
         ┌─┴┐  ┌──┴┐  ┌──┴─┐
         │12│  │ 21│  │ 48 │
         └┬┬┘  └┬┬─┘  └┬──┬┘
         ┌┘└┐  ┌┘└──┐ ┌─┘  └─┐
        ┌┴┐┌┴┐┌┴─┐┌┴────┐┌───┴┐┌┴─┐
        │10││15││20││24 30 ││45 47││50 52│
        └──┘└──┘└──┘└──────┘└─────┘└─────┘

删除目标:10
```

**Step 1 删 10 & 合并**:10 所在 [10] 兄弟 [15] 不够借,合并——父 [12] 的 12 下借,与 [15] 合并成 [12, 15]。结果:

```
父 [12] → 空(keys 减到 0!)
左子 [10] → (删除)
右子 [15] → 合并后变 [12, 15]
```

**Step 2 父 [12] 结点空 → 触发向上合并**:父的兄弟是 [21],父 [12] 和 [21] 的父结点是根 [18, 33]。父 [12] 需要与兄弟 [21] 合并——根的 18 下借,[12 空] + 18 + [21] 合并成 [18, 21]。结果:

```
根 [18, 33] → [33](keys 减 1)
原父 [12] → (删除)
兄弟 [21] → 合并后变 [18, 21]
```

**Step 3 根 [33] 仍满足 1≤keys≤2(根结点至少 1 key)→ 结束**。

**最终结果**(本例根的 keys 减到 1 但仍合规;若根 keys 减到 0,则会**直接删根,树高 -1**,这就是 2023 真题的考查核心):

```
          ┌────┐
          │ 33 │
          └┬──┬┘
       ┌──┘   └──┐
     ┌─┴─┐    ┌──┴─┐
     │18 21 │    │ 48 │
     └┬─┬┬─┘    └┬──┬┘
    ┌─┘ │ └──┐   │   │
   [12 15]│[24 30][45 47][50 52]
           [20]
```

⚠️ **易错点(2023 真题)**:合并可能导致父结点关键字减少;若父为根且关键字减到 0,**直接删根,树高 −1**(与插入分裂的"树高 +1"对称)。这是 B 树唯一能让树高变化的两种情形。

### 【§二 收束小结】

到这里你已经掌握了 B 树的全部要点:① 阶 m 的含义;② 5 条核心性质;③ 结点结构 `n | P₀ | K₁ | P₁ | ... | Kₙ | Pₙ` 与"子比键多一";④ 查找(两基本操作,磁盘 I/O + 内存比较);⑤ 高度上下界 $\lceil \log_m(n+1) \rceil \le h \le \lfloor \log_{\lceil m/2 \rceil} \tfrac{n+1}{2} \rfloor + 1$;⑥ 插入三步走与连锁分裂至根;⑦ 删除 3 情况(直接删 / 父子换位 / 合并)与合并连锁至根。B 树在磁盘查找方面已经非常优秀,但数据库场景还有两个特殊需求——**① 所有查找的 I/O 次数要一致(查什么都走同样多的层);② 支持范围查询**(SQL 的 `WHERE x BETWEEN a AND b`)。这就催生了 B 树的变体——**B+ 树**。

## 三、B+ 树 —— 数据库专用的 B 树变体

### 1. 应用场景:为什么数据库选 B+ 树(2017 真题)

**B+ 树**是应**数据库所需**而出现的一种 B 树的变形树(王道原文 2017 命题追踪)。两个核心优势让它成为数据库引擎的首选:

- **查找 I/O 次数绝对一致** —— 每次查找无论 key 命不命中,都必须从根走到叶。这让数据库查询的延迟**可预测**,便于做性能规划和 SLA 承诺
- **支持范围查询** —— 所有叶结点串成一个链表,找到起点叶后沿链表顺序遍历就能完成范围查询,不需要重复从根出发

**工程实例**:MySQL InnoDB 的主键索引、Oracle 的 B-tree Index、PostgreSQL 的 B-tree Index 都是 B+ 树。你写的每一句 `SELECT * FROM table WHERE id BETWEEN 100 AND 200`,底层都在调用 B+ 树的叶链表顺序扫描。

### 2. B+ 树的 5 条定义

一棵 m 阶的 B+ 树应满足**下列条件**(王道原文):

1. 每个**分支结点**最多有 m 棵子树(孩子结点)
2. **非叶根结点**至少有 2 棵子树,**其他每个分支结点**至少有 ⌈m/2⌉ 棵子树
3. 结点的**子树个数 = 关键字个数**(⚠️ 与 B 树不同!B 树是子树 = 关键字 + 1)
4. **所有叶结点**包含**全部关键字**及**指向相应记录的指针**,叶结点中将关键字按大小顺序排列,并且**相邻叶结点按大小顺序相互链接起来**(支持顺序查找)
5. **所有分支结点**(可视为**索引的索引**)中仅包含它的各个子结点(下一级的索引块)中**关键字的最大值**及**指向其子结点的指针**

⚠️ **核心差异点**:B+ 树的分支结点**不对应记录存储地址**——分支结点只是索引,真正的数据记录全部集中在叶结点。这是 B+ 树和 B 树最本质的差异。

### 3. B+ 树的结点结构对比

B+ 树的结点结构与 B 树**看起来很像**,但关键字数和子树数的关系**完全不同**:

```
【B 树结点】                      【B+ 树结点】
┌──┬──┬──┬──┬──┬──┬─┬──┬──┐     ┌──┬──┬──┬──┬──┬──┬─┬────┬────┐
│n │P₀│K₁│P₁│K₂│P₂│…│Kₙ│Pₙ│     │n │P₀│K₀│P₁│K₁│P₂│…│Pₙ₋₁│Kₙ₋₁│
└──┴──┴──┴──┴──┴──┴─┴──┴──┘     └──┴──┴──┴──┴──┴──┴─┴────┴────┘
 n keys → n+1 个子树              n keys → n 个子树
 K 从 1 开始编号                    K 从 0 开始编号
 每个 Kᵢ 是分隔 key                 每个 Kᵢ 是对应 Pᵢ 所指子树的**最大 key**
```

⚠️ **考试最易错点(2016 真题精准命中)**:
- B 树:**n 关键字 ↔ n+1 子树**(关键字切分 n+1 个区间)
- B+ 树:**n 关键字 ↔ n 子树**(每个 key 是子树的最大值,与子树 1:1 对应)

很多同学把两种树的结构记混,做题时遇到"结点内有 n 个关键字,有几个子树指针",要先确定树的种类。

### 4. 4 阶 B+ 树完整实例(图 7.32)

王道图 7.32 展示了一棵 4 阶 B+ 树,设记录数据为 8、10、16、20、40、50、55、60、69、77、80、85:

```
     ┌─ 根指针
     │                  ┌────┬────┐
     ↓                  │ 60 │ 85 │                  ← 分支结点(索引的索引)
                        └─┬──┴──┬─┘                    仅存子树最大 key
                   ┌──────┘     └─────┐
             ┌─────┴─────┐       ┌────┴─────┐
             │10│20│50│60│       │ 77 │ 85 │
             └┬──┬──┬──┬┘       └─┬────┬──┘
              │  │  │  │           │    │
       ┌──────┴┐┌┴─┐┌┴┐ ┌┴─┐ ┌────┴───┐┌┴──────┐
       │8 │ 10││16 20││40 50││55 60│ │69 │ 77 ││80 │ 85 │
       └─┬─┬──┘└──┬──┘└──┬──┘└──┬──┘ └──┬─┬──┘└──┬─┬──┘
         │ │      │      │      │        │ │     │ │
       ↓记录↓    ↓记录↓  ↓记录↓  ↓记录↓     ↓记录↓  ↓记录↓
       (所有叶结点按大小串成链表,如 8→10→16→20→40→50→...→85)
     ─ 最小叶指针
```

两种关键指针:
- **根指针**:指向 B+ 树的根结点,支持从根的多路查找
- **最小叶指针**:指向值最小的叶结点,支持从最小叶开始的顺序查找

### 5. B+ 树的查找方式(两种路径)

因为 B+ 树的叶层既是有序数组又是链表,它支持**两种查找**:

#### 方式 A:从根多路查找(类似 B 树)

```c
// B+ 树多路查找伪代码
LeafNode *BPlus_Search(BPlusTree T, KeyType key){
    BPlusNode *p = T->root;
    while(p is 分支结点){
        // 在分支结点内找最小的 Kᵢ ≥ key
        int i = 0;
        while(i < p->n && p->key[i] < key) i++;
        p = p->ptr[i];                       // 向下走一次 I/O
    }
    // 此时 p 是叶结点
    // 在叶结点内找等于 key 的位置
    return search_in_leaf(p, key);
}
```

**关键差异**:非叶结点的关键字**等于给定值时不终止**,继续向下查到叶!因为非叶结点的 key 只是"子树最大值",真实数据只在叶里。所以**每次查找都是从根到叶的完整路径**,I/O 次数 = 树高(固定),这是 B+ 树性能可预测的根源。

#### 方式 B:从最小叶顺序查找

直接从"最小叶指针"出发,沿叶链表顺序扫描。这是 B+ 树对**范围查询**的原生支持,也是 SQL 如 `WHERE x BETWEEN a AND b` 的底层实现方式——先用方式 A 定位起点叶,再用方式 B 扫到终点。

⚠️ **易错点**:方式 A 的"不在中间结点终止"是与 B 树最重要的**查找行为差异**。B 树可能在内部结点命中(查到就返回),B+ 树必须到叶。做题时看清楚是 B 树查找还是 B+ 树查找。

### 6. B 树 vs B+ 树 五大差异(2016 真题考查核心)

m 阶 B 树与 m 阶 B+ 树的主要差异汇总(融合王道 5 条 + 零壹讲义对比表):

| 对比维度 | B 树 | B+ 树 |
|---|---|---|
| **① 关键字与子树数关系** | n 关键字 → **n + 1 子树** | n 关键字 → **n 子树** |
| **② 非叶结点关键字数范围** | ⌈m/2⌉ − 1 ≤ n ≤ m − 1(非根);根 1 ≤ n ≤ m − 1 | ⌈m/2⌉ ≤ n ≤ m(非根);根 2 ≤ n ≤ m |
| **③ 叶结点内容** | **不包含信息**(虚构失败结点,指针为空) | **包含全部关键字**,及指向记录的指针,并组织成链表 |
| **④ 非叶结点内容** | 包含关键字 + 指向记录的存储地址(每个 key 对应一条记录) | **仅索引作用**——只含各子树最大 key + 指针,**不对应记录存储地址** |
| **⑤ 查找行为** | 可能在非叶结点就找到 key,**长度不一定** | 每次查找都是**从根到叶的完整路径**,长度**固定** |

**为什么这些差异让 B+ 树更适合数据库?**

- **差异 ③ + ④**:所有数据记录集中在叶层,分支结点只存索引——这让每个磁盘块能装下更多索引项(因为不用带记录),**同样数据量下 B+ 树更矮**,I/O 次数更少
- **差异 ⑤**:查找路径固定 → 性能可预测,便于数据库做查询优化(统计平均响应时间、预估 I/O)
- **差异 ③**:叶层链表 → **范围查询天然支持**,不需要多次从根走路

### 【§三 收束小结】

到这里你已经掌握了 B+ 树的全貌:① 数据库专用动机;② 5 条定义(与 B 树的关键差异在"子=键"、"叶含数据+链表"、"非叶仅索引");③ 结点结构 `n | P₀ | K₀ | P₁ | K₁ | ...` 与 B 树错位不同;④ 双指针支持两种查找方式;⑤ 5 大差异精准命中 2016 真题。两种树已经全部登场,下一节我们把它们放一起做终极对比,同时把整个第 7 章的"平衡力度滑尺"画完整。

| **B+ 树定义** | B 树变体,专为**数据库**设计 |
| **B+ 树 5 定义** | ①≤m 子;②非叶根≥2 子,其他≥⌈m/2⌉ 子;③**#子=#keys**;④叶含全部数据+链表;⑤分支仅存子最大 key |

### 4. 考场一分钟速查(考前 30 秒扫一眼)

```
┌─── B 树 ───┐      ┌── B+ 树 ──┐
│阶 m=最大子 │      │数据库专用 │
│平衡因子=0 │      │叶=全部数据│
│#子=#K+1 ⚠│      │#子=#K ⚠   │
│叶=失败结点│      │叶=真数据  │
│查找不定长 │      │查找必到叶 │
└───────────┘      └───────────┘

【三大必背公式】
  h_min = ⌈log_m(n+1)⌉           ← 每结点塞满
  h_max = ⌊log_⌈m/2⌉((n+1)/2)⌋+1 ← 每结点塞最少
  失败结点数 = n + 1                ← 每 n 个 key 对应 n+1 外部叶

【插入/删除的连锁】
  插入溢出 → 分裂 ⌈m/2⌉ 上提 → 父也溢出则连锁 → 根也分裂则树高+1
  删除空位 → 兄弟够借?父子换位(父下兄上)
                    ↓ 否
           兄弟不够借?合并(兄+父key) → 父少key → 根空则删根树高-1

【五大易错 ⚠】
  ⚠ B 树叶结点 = 虚构失败结点,不是真数据
  ⚠ 分裂位置 = ⌈m/2⌉ 向上取整,不是 m/2
  ⚠ B 树 #子 = #K+1,B+ 树 #子 = #K  
  ⚠ B+ 树查找必到叶,B 树可能在非叶命中
  ⚠ 合并使父 key 减 1,根空则删根 → 树高 -1

【父子换位法】(B 树删除最易画错)
  兄弟够借情景:
  ① 父的分隔 key 下移到被删位置
  ② 兄弟的最大(左兄弟)/最小(右兄弟)key 上移到父
```
